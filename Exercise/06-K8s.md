## Kubernetes / EKS

Then it moved into Kubernetes. They wanted to know my EKS experience specifically, not just Kubernetes in general. If you've only worked with AKS or GKE, make sure you understand where EKS differs — how IAM integrates with service accounts (IRSA), node group management, and networking with different CNI options.

**Answer in simple terms:**

EKS is just Kubernetes managed by AWS. AWS runs the control plane (the brain), you manage the worker nodes (the machines that run your apps).

- **IRSA (IAM Roles for Service Accounts):** Normally, your app inside Kubernetes has no idea about AWS permissions. IRSA lets you say "this specific app in my cluster is allowed to access this specific S3 bucket." It connects a Kubernetes identity to an AWS permission — without storing any passwords or keys.

- **Node groups:** Think of these as pools of machines. You might have one group of big machines for heavy workloads and another group of small cheap machines for background jobs. EKS lets you manage these pools — scale them up, scale them down, upgrade them.

- **CNI (Container Network Interface):** This is how pods get their IP addresses. In EKS, the default CNI (AWS VPC CNI) gives each pod a real IP address from your VPC. That means pods can talk to other AWS resources (like RDS or ElastiCache) directly, no extra hops. Other CNIs like Calico use overlay networks — virtual IPs on top of the real network.

## Argo CD

Argo CD was a big focus. Not just "have you used it" but how did you actually deploy it. How can you migrate Argo CD from Cluster A to Cluster B:

- Export everything from Cluster A — manifests, secrets, repo credentials. Nothing left behind.
- Provision Argo CD fresh on Cluster B using the same Helm values.
- Restore everything. Watch the apps sync. Wait until everything is green.

**Answer in simple terms:**

Argo CD is a tool that watches your Git repo and makes sure your Kubernetes cluster matches what's in Git. You push a change to Git, Argo CD sees it and applies it to the cluster automatically. That's GitOps — Git is the single source of truth.

**How to migrate Argo CD between clusters:**

Think of it like moving to a new house.

1. **Pack everything from the old house** — take a backup of all Argo CD app definitions, secrets (repo passwords, credentials), and any custom config.
2. **Set up the new house** — install Argo CD on the new cluster using the same Helm chart and values file so it's configured identically.
3. **Unpack** — restore the backup. Point it at the same Git repos.
4. **Check** — Argo CD reads Git, compares it to the new cluster, and syncs everything. When all apps show "Healthy" and "Synced," you're done.

## EKS from Scratch

The question I found most interesting was this: you're given a completely blank AWS account, walk me through getting to a running EKS cluster with Argo CD on it. They want your actual thought process. Account baseline, IAM, VPC and subnets, cluster provisioning, node groups, kubeconfig setup, then Argo CD installation and Git repo connection. Don't skip the networking part — that's where they're listening most carefully.

**Step-by-step in plain English:**

1. **Account baseline** — Enable CloudTrail (audit log), set up an S3 bucket for state files, create an admin IAM role. Basically, make the account usable and secure.

2. **IAM** — Create roles: one for the EKS cluster itself, one for the worker nodes, and one for you to manage it. Each role gets only the permissions it needs.

3. **VPC and networking** — Create a VPC (your private network). Add at least 2 public subnets (for load balancers) and 2 private subnets (for your worker nodes) across different availability zones. Set up a NAT gateway so private nodes can pull container images from the internet without being exposed.

4. **EKS cluster** — Create the cluster using `eksctl` or Terraform. This gives you the control plane — the brain that manages everything.

5. **Node groups** — Add worker machines to the cluster. These are EC2 instances that actually run your containers. Pick instance types based on your workload.

6. **kubeconfig** — Run `aws eks update-kubeconfig` to get your local machine connected to the cluster. Now `kubectl` commands work.

7. **Argo CD** — Install it via Helm. Expose the UI. Connect it to your Git repo. Create your first Application resource pointing at a Git path. Argo CD takes over from here.

## Rolling Update vs Canary

Rolling gradually replaces old pods with new ones, all traffic shifts as it goes. Canary sends a small percentage of traffic to the new version first, you validate it, then shift the rest.

**Answer in simple terms:**

Imagine you have 10 waiters in a restaurant and you want to replace their uniforms.

- **Rolling update:** You pull one waiter out at a time, give them the new uniform, and send them back. Slowly, all 10 get new uniforms. Customers barely notice. If the new uniform has a problem, some waiters already have it. Kubernetes does this by default with Deployments.

- **Canary:** You give the new uniform to just 1 waiter first. Watch how it goes. Customers complain about the color? Stop right there — only 1 waiter is affected. Everything fine? Roll it out to the rest. You need a tool like Argo Rollouts or Istio to do this in Kubernetes — plain Deployments don't support it natively.

## ALB vs NLB

NLB is Layer 4: TCP/UDP, ultra-low latency, static IPs, good for databases or internal services.
ALB is Layer 7: understands HTTP, supports path-based routing and host headers, better for web apps and microservices.

**Answer in simple terms:**

Think of a post office.

- **NLB (Network Load Balancer)** — It's the mailman. He picks up the package and delivers it as fast as possible. He doesn't open it, doesn't read the label, doesn't care what's inside. Fast, dumb, reliable. Use it when your app speaks raw TCP — databases, gRPC, game servers, or anything that needs static IPs and very low latency.

- **ALB (Application Load Balancer)** — It's the front desk clerk. She reads the label, sees it says "accounting department," and routes it to the right floor. She understands HTTP — she can look at the URL path (`/api` vs `/web`) or the hostname (`api.example.com` vs `web.example.com`) and send traffic to different backend services. Use it for web apps, REST APIs, and microservices.

| | NLB | ALB |
|---|---|---|
| Layer | 4 (TCP/UDP) | 7 (HTTP/HTTPS) |
| Routing | By port/IP | By path, host, headers |
| Latency | Ultra-low | Slightly higher |
| Static IP | Yes | No (use Global Accelerator) |
| Best for | Databases, gRPC, internal | Web apps, APIs, microservices |

## Linux Basics

They finished with Linux basics. Creating a user, generating SSH keys, the right permissions on `.ssh`. Simple stuff but people fumble it under pressure, so just make sure it's automatic for you.

**Quick reference:**

```bash
# Create a user
sudo useradd -m -s /bin/bash devuser

# Set a password
sudo passwd devuser

# Generate SSH key pair
ssh-keygen -t ed25519 -C "devuser@company.com"

# Set up SSH for the user
sudo mkdir -p /home/devuser/.ssh
sudo cp ~/.ssh/id_ed25519.pub /home/devuser/.ssh/authorized_keys

# Fix permissions — this is where people mess up
sudo chown -R devuser:devuser /home/devuser/.ssh
sudo chmod 700 /home/devuser/.ssh
sudo chmod 600 /home/devuser/.ssh/authorized_keys
```

**Why these permissions matter:** SSH refuses to work if the `.ssh` directory or `authorized_keys` file is readable by others. `700` on the directory means only the owner can enter it. `600` on the file means only the owner can read it. Get these wrong and SSH silently rejects the key — no error, just "permission denied."
